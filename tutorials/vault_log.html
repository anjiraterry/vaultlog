Before ownership, borrowing, or lifetimes make sense ; you need to understand where values actually live in memory. This is the bedrock. Everything else is built on it.

JavaScript Hides This. Rust Doesn't.

In JavaScript, you never think about memory. The engine allocates it, and the garbage collector frees it. In Rust, you are always ; implicitly ; making decisions about where data lives. The compiler enforces those decisions.

What a String Actually Is

This is the most important memory diagram in Rust. A String is not just text ; it's a three-field struct on the stack that points to actual text data on the heap:

let description = String::from("Purchased 123 Main St")

Why does this matter? Because when you move or drop a String, three things happen: the stack struct is cleared, the heap data is freed. This is the whole reason ownership exists ; to track exactly who is responsible for freeing that heap allocation.

🔑 The Core Insight

Types that live entirely on the stack (i32, bool, char) can be copied cheaply ; just duplicate a few bytes. Types that own heap data (String, Vec) cannot be copied trivially ; they'd create two pointers to the same heap memory, and when both tried to free it, you'd have a double-free crash. This is exactly why Rust invented ownership and move semantics.

◆ OWNERSHIP

One Owner,
Always

Ownership is Rust's core innovation. It's a set of three compile-time rules that replace garbage collection entirely ; with zero runtime cost.

The Three Ownership Rules

THE RULES ; memorise these

  • Rule 1: Every value has exactly one owner at any moment
  • Rule 2: There can only be one owner at a time
  • Rule 3: When the owner goes out of scope, the value is dropped (freed)

That's it. The entire borrow checker and lifetime system exists to enforce these three rules. Everything else is a consequence of them.

Scope is the Owner's Lifetime

{   // scope begins
    let description = String::from("Purchased 123 Main St");
    //  ^^^^^^^^^^^  owns the heap allocation

    // `description` is valid here, we can use it
    println!("{}", description);

}   // scope ends ; Rust calls drop(description)
    // heap memory is freed HERE, automatically
    // no garbage collector. no free(). just scope.

🔁 JS Translation

In JavaScript, the GC watches for when no references exist and then (eventually) frees the memory. In Rust, freedom happens immediately and deterministically at the closing brace. No background process, no unpredictable pauses. This is why Rust is used in systems programming ; you know exactly when memory is freed.

Ownership in vaultlog ; the LogEntry

Look at LogEntry ; every field with type String means the struct owns that heap data:

pub struct LogEntry {
    pub id:            String,  // LogEntry OWNS this heap string
    pub description:   String,  // LogEntry OWNS this heap string
    pub previous_hash: String,  // LogEntry OWNS this heap string
    pub hash:          String,  // LogEntry OWNS this heap string
    pub category:      EntryCategory, // owned enum value (stack)
}
// When a LogEntry is dropped, ALL five heap allocations are freed.
// Rust generates this cleanup code automatically from the struct definition.

Drop Order ; Rust is Precise

When a struct is dropped, Rust drops its fields in reverse declaration order. This is deterministic and guaranteed. Your LogEntry frees hash first, then previous_hash, then description, then id.

What "Out of Scope" Actually Means

Scope in Rust is defined by curly braces { }. Every block ; function body, if branch, loop body, match arm ; is a scope. Variables are dropped when their enclosing scope closes:

fn add_entry(description: String) {  // `description` enters scope here
    let entry = LogEntry {
        id: uuid::Uuid::new_v4().to_string(),
        description,   // ownership MOVES into LogEntry
        // ...
    };
    // entry is valid here
    append_entry(&entry);  // we BORROW entry (next chapter)
}   // entry drops here → ALL its String fields freed
    // description was already moved into entry, so only freed once
◆ MOVE SEMANTICS

Why Values
Disappear

Move semantics is where most Rust beginners hit their first wall. The compiler rejects code that looks completely normal. Here's exactly why ; and why it's protecting you.

The Problem Move Solves

Imagine two variables pointing to the same heap allocation. When both try to free that memory when they go out of scope ; that's a double-free error. In C, it's undefined behaviour. In JavaScript, the GC tracks references to prevent it. In Rust, the compiler makes it impossible to compile.

WHY "COPY BOTH" IS DANGEROUS ; the double-free problem
IF RUST COPIED s1 → s2
s1.ptr→ 0x2c9a00

s2.len8

Move = Transfer of Ownership

Instead of copying the heap data, Rust moves ownership. The original variable becomes invalid ; the compiler prevents you from using it ever again. This is checked at compile time, not runtime:

COMPILE ERROR ; use after move
let s1 = String::from("vaultlog");
let s2 = s1; // s1 MOVES into s2

println!("{}", s1); // ✗ ERROR!
// error[E0382]: borrow of moved
// value: `s1`
CORRECT ; use the new owner
let s1 = String::from("vaultlog");
let s2 = s1; // s1 moves to s2

println!("{}", s2); // ✓ s2 is the owner now
// s1 is gone ; compiler enforces this
AFTER MOVE ; s1 is invalidated on the stack
STACK
s1.ptr(moved ; unusable)
s1.len(moved ; unusable)
s2.ptr→ 0x2c9a00
s2.len8

HEAP
0x2c9a00"vaultlog"
↑ freed exactly once, by s2

Move into Functions ; the Most Common Surprise

Passing a value into a function moves ownership into that function. After the call, your variable is gone:

fn print_entry(entry: LogEntry) {  // takes OWNERSHIP of entry
    println!("{}", entry.description);
}   // entry drops here ; freed inside the function

fn main() {
    let entry = LogEntry { /* ... */ };
    print_entry(entry);  // entry MOVES into print_entry

    println!("{}", entry.description); // ✗ ERROR: use of moved value
}
// FIX: either return entry back, clone it, or borrow it (&entry)

Clone ; Explicit Deep Copy

If you genuinely need two independent copies, you call .clone(). This duplicates the heap data too. It's explicit because it's expensive ; Rust wants you to see every heap allocation:

let entry1 = LogEntry { /* ... */ };
let entry2 = entry1.clone();  // deep copy ; new heap allocations

// Both are valid, both own their data independently
println!("{}", entry1.description);  // ✓
println!("{}", entry2.description);  // ✓
// Requires #[derive(Clone)] on LogEntry

Copy Types ; the Exception

Simple stack-only types implement the Copy trait and are automatically copied on assignment. No move, no loss of original:

let x: i32 = 42;
let y = x;         // COPIED (not moved) ; x is still valid!
println!("{} {}", x, y);  // ✓ both work: 42 42

// Why? i32 is 4 bytes on the stack. There's no heap to worry about.
// Copying 4 bytes is so cheap Rust just does it automatically.

🔁 JS Parallel

In JavaScript, objects are always passed by reference ; two variables can point to the same object. When you write const b = a for an object, both a and b point to the same thing. This is convenient but causes subtle bugs. Rust's move semantics make ownership explicit ; there is always exactly one owner, and the compiler tells you when you violate that.

◆ BORROWING

& and &mut
Lending Without Giving

Moving ownership every time you pass a value to a function would make code unworkable. Borrowing is how you let functions use a value temporarily without transferring ownership.

The & Reference ; Immutable Borrow

Putting & in front of a type creates a reference ; a pointer to the value that doesn't take ownership. The original variable remains the owner:

fn compute_hash(entry: &LogEntry) -> String {
    //              ^ borrows entry ; does NOT take ownership
    let content = serde_json::to_string(entry).unwrap();
    // entry is used here...
}   // entry is NOT dropped here ; we never owned it!

fn main() {
    let entry = LogEntry { /* ... */ };
    let hash = compute_hash(&entry); // lend entry ; keep ownership
    println!("{}", entry.description); // ✓ still valid!
}
BORROWING ; a reference on the stack points to owned data

The &mut Reference ; Mutable Borrow

To modify a value through a reference, use &mut. But there's a strict rule: only one mutable reference at a time, ever:

fn finalise_entry(entry: &mut LogEntry, hash: String) {
    // Can modify because we have &mut
    entry.hash = hash;
}

fn main() {
    let mut entry = LogEntry { /* ... */ };
    //  ^^^ the variable itself must be `mut` too

    let hash = compute_hash(&entry);
    finalise_entry(&mut entry, hash); // mutable borrow
}

Why &str and &[T] Are References

A &str is literally a reference to a slice of a String's heap data ; a pointer plus a length. This is why you can take a &str of a String without copying anything:

let owned: String = String::from("Purchased 123 Main St");
let slice: &str = &owned;        // borrows the whole string as a slice
let part:  &str = &owned[0..9]; // borrows just "Purchased"
// No allocation happened ; slice/part are just (pointer, length) on the stack

// In vaultlog functions, take &str not String when you only need to read:
fn log_description(desc: &str) {  // ← efficient, no ownership needed
    println!("Entry: {}", desc);
}
◆ BORROW RULES

The Three
Borrow Laws

The borrow checker enforces three rules that make Rust's memory safety possible. Violate any one and your code won't compile. Understand all three and the checker becomes a tool, not a wall.

THE THREE BORROW RULES

  • Law 1: You can have any number of immutable (&) references ; OR
  • Law 2: You can have exactly one mutable (&mut) reference ; BUT
  • Law 3: You cannot have both at the same time

Law 1 ; Many Immutable Readers ✓

let entries: Vec<LogEntry> = read_all_entries().unwrap();

// Multiple & references are fine ; they all just READ
let r1 = &entries;
let r2 = &entries;
let r3 = &entries;
println!("{} {} {}", r1.len(), r2.len(), r3.len()); // ✓ all valid
// Why is this safe? None of them can modify entries, so no data races.

Law 2 ; One Exclusive Writer ✓

let mut entries: Vec<LogEntry> = Vec::new();

let m = &mut entries;   // one mutable reference
m.push(new_entry);     // ✓ can modify through m
// m goes out of scope after this block, releasing the mutable borrow

// NOW you can create another reference:
let r = &entries;       // ✓ fine ; m is no longer active

Law 3 ; Can't Mix Them ✗

COMPILE ERROR ; mixed borrows
let mut entries = load_entries();

let first = &entries[0]; // immutable borrow

entries.push(new_entry); // ✗ mutable borrow
// ERROR: cannot borrow as mutable
// because it is also borrowed as immutable

println!("{}", first.description); // first used here
FIXED ; borrows don't overlap
let mut entries = load_entries();

// Use first, then drop the reference:
{
    let first = &entries[0];
    println!("{}", first.description);
}   // first borrow ends here

// Now we can mutate safely:
entries.push(new_entry); // ✓

Why These Rules Exist ; The Real Reason

These aren't arbitrary restrictions. They are the compile-time equivalent of preventing data races. A data race happens when:

  • Two or more threads access the same memory
  • At least one is writing
  • There's no synchronisation

The borrow rules eliminate data races by making the pattern impossible to compile ; even in single-threaded code. This is also why Rust's concurrency story is so strong: the same rules that prevent bugs in single-threaded code prevent race conditions in multi-threaded code automatically.

The Borrow Checker in vaultlog's verify_chain

The chain verification function is a perfect example of correct borrowing:

pub fn verify_chain(entries: &[LogEntry]) -> bool {
    //               ^ borrows a SLICE ; doesn't own the Vec
    //               &[T] = borrowed view of contiguous data

    for i in 1..entries.len() {
        let prev = &entries[i - 1]; // borrow element i-1
        let curr = &entries[i];     // borrow element i
        // Two immutable borrows at once ; totally fine (Law 1)

        if prev.hash != curr.previous_hash {
            return false;
        }
    }
    true  // implicit return ; no semicolon
}

// Called as: verify_chain(&entries)
// The caller keeps ownership of entries. This function just reads.
◆ LIFETIMES

'a and Why They Exist

Lifetimes are the most intimidating Rust syntax. But they're not doing anything new ; they're just making explicit what the compiler already tracks invisibly. Once you see what problem they solve, the syntax becomes obvious.

The Problem: Dangling References

A dangling reference is a pointer to memory that has already been freed. In C, this is a silent crash waiting to happen. Rust's lifetime system makes dangling references a compile error:

DANGLING REFERENCE ; compile error
let reference;
{
    let entry = LogEntry { /* ... */ };
    reference = &entry;  // borrow entry
}   // entry DROPS HERE ; freed!

// ERROR: `entry` does not live long enough
println!("{}", reference.description);
// reference now points to freed memory
VALID ; reference lives within owner
let entry = LogEntry { /* ... */ };
{
    let reference = &entry;
    println!("{}", reference.description);
    // ✓ reference used while entry lives
}   // reference drops first, entry drops after
// safe: reference never outlives what it points to

Lifetime Annotations ; Naming How Long

Most of the time, the compiler infers lifetimes automatically (called lifetime elision). You only write explicit lifetime annotations when the compiler needs help connecting multiple references in a function signature.

Lifetime syntax looks like generic parameters but with a tick: 'a. You can use any name ; 'a, 'b, 'entry, 'static.

// Without explicit lifetimes ; compiler infers
fn get_description(entry: &LogEntry) -> &str {
    &entry.description
}
// Compiler reads this as: the output &str lives at LEAST as long as &entry
// It figures this out because there's only one input reference

// When there are MULTIPLE input references, you must be explicit:
fn longest_description<'a>(
    a: &'a LogEntry,   // 'a says: a's lifetime is named 'a
    b: &'a LogEntry,   // b has the SAME lifetime 'a
) -> &'a str {         // output lives as long as the shorter of a, b
    if a.description.len() > b.description.len() {
        &a.description
    } else {
        &b.description
    }
}

What 'a Actually Means

A lifetime annotation doesn't change how long something lives. It's a constraint you're expressing to the compiler ; "I promise the output reference is valid for at least as long as the input." If your code violates that promise, it won't compile.

LIFETIME SCOPES ; how the compiler thinks
fn main() {
let entry_a = LogEntry { .. };
'a begins
{
let entry_b = LogEntry { .. };
'b begins
let result = longest(&entry_a, &entry_b);
} // entry_b drops
'b ends ; result can no longer be used!
} // entry_a drops
'a ends
lifetime 'a (entry_a)
lifetime 'b (entry_b)

'static ; Lives Forever

There's one special lifetime: 'static. It means the reference is valid for the entire program. String literals are 'static because they're baked into the binary:

let s: &'static str = "genesis"; // lives for the whole program
// String literals are &'static str ; they live in the program binary
// The const LEDGER_FILE in storage.rs is &'static str:
const LEDGER_FILE: &'static str = "ledger.ndjson";
// (you can write &str here; compiler infers 'static for const)

Lifetimes in Structs

If a struct holds a reference (not an owned value), it needs a lifetime annotation:

// LogEntry owns its Strings, so NO lifetime needed:
pub struct LogEntry { pub description: String }  // ✓ no 'a

// If we held a reference instead, we'd need one:
pub struct EntryView<'a> {
    pub description: &'a str, // borrows, doesn't own
}
// This says: EntryView can't outlive whatever &description points to
// In practice, prefer owned Strings in structs ; avoids this complexity

✅ Practical Advice

In vaultlog, you almost never write explicit lifetime annotations because your structs own their data (String, not &str). Lifetimes become necessary when you write functions that return references, or when structs hold references. Start by using owned types everywhere, and only reach for lifetime annotations when the compiler tells you to. Then read its error message carefully ; it almost always tells you exactly what annotation to add.

A trait defines a set of methods that a type must implement. It's Rust's equivalent of a TypeScript interface ; but more powerful, because they can have default method implementations and they power the entire generics system.

Traits as Contracts

When you write #[derive(Serialize, Debug, Clone)] on LogEntry, you're telling Rust: "auto-implement these three traits for this struct." Each trait is a contract:

Defining Your Own Trait

// Define the trait ; the contract
pub trait Hashable {
    // Required: any implementor MUST provide this
    fn compute_hash(&self) -> String;

    // Default: implementors CAN override, but don't have to
    fn is_valid_hash(&self, expected: &str) -> bool {
        self.compute_hash() == expected  // uses the required method
    }
}

// Implement the trait for LogEntry
impl Hashable for LogEntry {
    fn compute_hash(&self) -> String {
        let content = serde_json::to_string(self).unwrap();
        let mut hasher = Sha256::new();
        hasher.update(content.as_bytes());
        format!("{:x}", hasher.finalize())
    }
    // is_valid_hash is inherited ; no need to write it again
}

// Now you can call both methods on any LogEntry:
let valid = entry.is_valid_hash(&entry.hash);

Display ; the fmt::Display Trait

The Display trait controls how your type prints with {} in format strings. You implement it manually when you want a human-readable output:

use std::fmt;

◆ GENERICS

<T> and
Trait Bounds

Generics let you write code that works for multiple types without duplicating it. Trait bounds let you say "this generic T must be able to do at least this." Together, they make Rust's zero-cost abstractions possible.

The Problem Generics Solve

Without generics, you'd need a separate function for every type:

WITHOUT GENERICS ; duplication
fn first_entry(v: &Vec<LogEntry>) -> &LogEntry {
    &v[0]
}
fn first_string(v: &Vec<String>) -> &String {
    &v[0]
}
// Identical logic, duplicated for every type
WITH GENERICS ; one function
fn first<T>(v: &Vec<T>) -> &T {
    &v[0]
}
// Works for Vec<LogEntry>, Vec<String>,
// Vec<i32>, Vec<anything>

Trait Bounds ; Constraining T

A generic T with no bounds knows nothing about the type ; you can't call any methods on it. Trait bounds tell the compiler what T can do:

// T must implement Debug ; so we can print it
fn print_item<T: fmt::Debug>(item: &T) {
    println!("{:?}", item);  // ✓ Debug guarantees this works
}

// Multiple bounds with + syntax
fn save_entry<T: Serialize + fmt::Debug>(entry: &T) {
    println!("{:?}", entry);               // needs Debug
    serde_json::to_string(entry).unwrap(); // needs Serialize
}

// Equivalent with `where` clause (cleaner for complex bounds):
fn save_entry<T>(entry: &T)
where
    T: Serialize + fmt::Debug
{
    /* same body */
}

Generics in vaultlog ; Vec<LogEntry>

Vec<T> is the most common generic you'll use. It's a growable heap-allocated array that works for any type:

// Vec<LogEntry> ; a growable list of LogEntry values
pub fn read_all_entries() -> Result<Vec<LogEntry>, Box<dyn std::Error>> {
    let file = std::fs::File::open(LEDGER_FILE)?;
    let reader = BufReader::new(file);

    // .lines() returns Iterator<Item = Result<String>>
    // .filter_map() is generic: works on any Iterator
    let entries: Vec<LogEntry> = reader
        .lines()
        .filter_map(|line| line.ok())              // Result → Option
        .filter_map(|line| serde_json::from_str(&line).ok()) // parse JSON
        .collect();  // <-- collect() is generic: "gather into a collection"
                     // type annotation tells it WHICH collection to build
    Ok(entries)
}

Box<dyn Error> ; Dynamic Dispatch

You've seen this in return types. It's a specific use of generics worth understanding:

// Box<dyn Error> means:
// Box = heap allocation (the error lives on the heap)
// dyn = dynamic dispatch (the exact type is unknown at compile time)
// Error = the trait the value implements

pub fn append_entry(entry: &LogEntry) -> Result<(), Box<dyn std::Error>> {
    // This can return io::Error, serde_json::Error, or any other error
    // Box<dyn Error> is the "I don't care which error" escape hatch
    let mut file = OpenOptions::new()
        .create(true).append(true).open(LEDGER_FILE)?; // io::Error
    let line = serde_json::to_string(entry)?;           // serde error
    writeln!(file, "{}", line)?;                       // io::Error
    Ok(())
}

🔁 JS Translation

Generics in Rust are like TypeScript generics (function first<T>(arr: T[]): T) ; same concept, just stricter. The key difference: Rust generics with trait bounds are monomorphised at compile time ; the compiler generates a separate version of the function for each concrete type it's called with. No runtime overhead, unlike TypeScript generics which are erased.

All Five Concepts
in One File

Every concept from this guide appears together in the vaultlog source. Here's the full annotated storage module ; read it top to bottom and you'll see all five systems interacting in real code.

storage.rs ; Fully Annotated

// 1. TRAITS: these imports bring traits into scope.
//    BufRead, Write, Read are traits that add methods to file types.
use std::io::{BufRead, BufReader, Write};
use std::fs::OpenOptions;
use crate::models::LogEntry;

// LIFETIMES: &'static str ; this string lives for the program's lifetime
// OWNERSHIP: LEDGER_FILE is a const ; no owner, baked into the binary
const LEDGER_FILE: &str = "ledger.ndjson";

pub fn append_entry(
    entry: &LogEntry     // BORROWING: & ; we borrow, don't take ownership
                          // caller keeps the entry after this function returns
) -> Result<(), Box<dyn std::Error>>
// GENERICS: Result<T, E> is generic. T = () (unit, "no value on success")
// Box<dyn Error> = heap-allocated trait object, any error type
{
    let mut file = OpenOptions::new()
        .create(true)
        .append(true)
        .open(LEDGER_FILE)?;
    //              ^ OWNERSHIP: OpenOptions::open() returns Result<File>
    //                File is MOVED into `file` ; this fn owns it
    //              ? OPERATOR: if Err, converts and returns it early

    let line = serde_json::to_string(entry)?;
    //  GENERICS: to_string<T: Serialize> ; works on anything Serializable
    //  TRAITS: LogEntry must impl Serialize (from #[derive(Serialize)])
    //  OWNERSHIP: `line` is a new String, owned by this scope

    writeln!(file, "{}", line)?;
    //  TRAITS: writeln! uses the Write trait on `file`
    //  BORROWING: writes to `file` which we own via &mut (implicit in writeln!)

    Ok(())
    // STACK: () is zero bytes ; we're just signalling success
    // OWNERSHIP: file DROPS here ; OS file handle closed automatically
    // OWNERSHIP: line DROPS here ; its heap String freed
}   // entry was BORROWED ; it lives on, unchanged, in the caller

pub fn read_all_entries() -> Result<Vec<LogEntry>, Box<dyn std::Error>> {
    let file = std::fs::File::open(LEDGER_FILE)?;
    // OWNERSHIP: File is opened, moved into `file`

    let reader = BufReader::new(file);
    // OWNERSHIP: `file` MOVES into BufReader ; file is gone from this scope
    // TRAITS: BufReader implements BufRead, giving us .lines()

    let entries = reader
        .lines()
        // GENERICS: .lines() returns impl Iterator<Item = Result<String>>

        .filter_map(|line| line.ok())
        // CLOSURES: |line| is a closure ; like an arrow function
        // GENERICS: filter_map<B, F> where F: FnMut(Self::Item) → Option<B>
        // OWNERSHIP: `line` is moved into the closure

        .filter_map(|line| serde_json::from_str(&line).ok())
        // BORROWING: &line ; from_str borrows the string to parse it
        // GENERICS: from_str::<LogEntry> ; generic over Deserialize
        // TRAITS: LogEntry must impl Deserialize

        .collect::<Vec<LogEntry>>();
        // GENERICS: collect is generic ; turbofish syntax ::<> specifies the output type
        // OWNERSHIP: all LogEntry values are OWNED by the Vec
        // HEAP: Vec allocates on the heap to hold all entries

    Ok(entries)
    // OWNERSHIP: entries moves OUT of this function into the caller
}   // reader drops ; file handle closed. We don't drop entries ; it was returned.

The Ownership Flow of a Full Add Command

let description = String::from("Purchased 123 Main St")
description created ; owns heap allocation
let entry = LogEntry { description, .. }
description MOVES into LogEntry ; LogEntry now owns it
let hash = compute_hash(&entry)
entry is BORROWED ; compute_hash reads it, returns a new String
entry.hash = hash
hash (new String) MOVES into the entry struct field
append_entry(&entry)
entry BORROWED ; storage serialises it, writes to file
} ← closing brace of add_entry()
entry DROPS ; all its String fields freed. One allocation, one free.

✅ What to Do Next

Return to this guide whenever the compiler rejects your code. Ask: Is this an ownership error? A borrow error? A lifetime error? The compiler error message will contain the answer and often the fix. Over time, you'll stop needing this reference ; these rules become instinctive, and you'll start writing code that the compiler accepts on the first try.
