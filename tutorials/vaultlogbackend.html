PHASE 1 · CLI Fundamentals

Result<T, E>

The foundation of Rust's error handling. There is no try/catch — errors are values that your functions return explicitly.

In JavaScript, errors are thrown as exceptions. In Rust, errors are values that you handle like any other data. A function either succeeds with an Ok(value) or fails with an Err(error). This forces you to think about failure cases upfront.

The Result enum — two outcomes

enum Result<T, E> {
    Ok(T),   // Success — holds the value of type T
    Err(E),  // Failure — holds the error of type E
}

Any function that can fail returns Result<T, E>. T is the success type, E is the error type.

Comparing to JavaScript

JavaScript try/catch
async function readFile(path) {
  try {
    const data = 
      await fs.readFile(path);
    return data;
  } catch(err) {
    console.error(err);
    throw err;
  }
}
Rust Result
fn read_file(path: &str)
  -> Result<String, io::Error>
{
  let content = 
    fs::read_to_string(path)?;
  Ok(content)
}

Three ways to handle Result

1. The ? operator (most common)

// ? propagates errors up — if it's Err, return it
fn read_and_parse(path: &str) 
  -> Result<i32, Box<dyn std::error::Error>>
{
    let content = fs::read_to_string(path)?;  // if Err, return now
    let num = content.trim().parse::<i32>()?;  // if Err, return now
    Ok(num)                                    // success
}

2. Pattern matching (when you need custom handling)

match fs::read_to_string(path) {
    Ok(content) => {
        println!("Read {} bytes", content.len());
        Ok(content)
    }
    Err(err) if err.kind() == io::ErrorKind::NotFound => {
        // File doesn't exist — create it
        fs::write(path, "default")?;
        Ok("default".to_string())
    }
    Err(err) => Err(err.into()),
}

3. Methods on Result

.unwrap()Get value or panicIn tests and when you're certain it's Ok
.expect("msg")Get value or panic with messagePrototyping, or when panic is acceptable

Box<dyn Error> — accepting any error type

In vaultlog, your functions will return this:

fn append_entry(entry: &LogEntry)
  -> Result<(), Box<dyn std::error::Error>>
{
    let mut file = fs::OpenOptions::new()
        .create(true)
        .append(true)
        .open("ledger.ndjson")?;

    let line = serde_json::to_string(entry)?;
    writeln!(file, "{}", line)?;
    Ok(())  // explicit "success, no value"
}
🎯 Why Box<dyn Error>?

Box<dyn Error> is a "trait object" that can hold any error type — io::Error, serde_json::Error, parse errors, etc. The ? operator automatically converts them. This is the pragmatic choice for beginners: "I don't know what error types I'll encounter, just give me a box that holds them all."

PHASE 1 · Error Propagation

The ? Operator

The single most important syntax in Rust error handling. It means: "If this returns Err, immediately return that error from my function. If it's Ok, unwrap and continue."

? is syntactic sugar for early return

// Without ? — what ? is hiding
let file = match fs::OpenOptions::new().open(path) {
    Ok(f) => f,
    Err(e) => return Err(e),  // bail out with error
};

// With ? — one character
let file = fs::OpenOptions::new().open(path)?;

Chaining ? calls

This is where ? really shines. You can chain operations without nesting:

fn process_entry(path: &str) -> Result<LogEntry, Box<dyn std::error::Error>> {
    // Read file
    let json = fs::read_to_string(path)?;
    
    // Parse JSON
    let entry: LogEntry = serde_json::from_str(&json)?;
    
    // Validate timestamp is recent
    let age = Utc::now() - entry.timestamp;
    if age.num_days() > 7 {
        return Err("Entry is too old".into());
    }
    
    // Success
    Ok(entry)
}

Notice there's no nesting or callback hell. Each ? either succeeds and continues, or returns an error. Clean control flow.

? only works in functions that return Result (or Option)

// ✗ This won't compile
fn main() {
    let content = fs::read_to_string("file.txt")?;  // error! main returns ()
}

// ✓ This works
fn main() -> Result<(), Box<dyn std::error::Error>> {
    let content = fs::read_to_string("file.txt")?;  // ok!
    Ok(())
}

Converting error types with ?

The ? operator automatically calls .into() on the error to convert it to the return type. This is why Box<dyn Error> works so well — all standard errors implement the conversion:

fn mixed_errors() -> Result<i32, Box<dyn std::error::Error>> {
    // io::Error
    let content = fs::read_to_string("file.txt")?;
    
    // serde_json::Error
    let json: serde_json::Value = serde_json::from_str(&content)?;
    
    // std::num::ParseIntError
    let num = json["count"].as_str().ok_or("not a string")?.parse::<i32>()?;
    
    Ok(num)
}
⚠ The ? operator consumes the error

Once you use ?, the error is converted and propagated — you can't inspect it. If you need to handle the error differently based on its type, use match instead.

PHASE 1 · Control Flow

Pattern Matching

Rust's match is a supercharged switch statement. It's exhaustive — the compiler forces you to handle every possible case.

Basic match

JavaScript switch (fallthrough)
switch(action) {
  case "add":
    doAdd();
    break;  // forgot this?
  case "list":
    doList();
    break;
  default:
    doOther();
}
Rust match (exhaustive, no fallthrough)
match action {
    "add" => {
        do_add();
    }
    "list" => {
        do_list();
    }
    _ => {  // compiler requires catch-all
        do_other();
    }
}

Matching enums — extracting data

This is where match becomes incredibly powerful. Each enum variant can hold different data, and match lets you extract it:

enum Command {
    Add { description: String, category: String },
    List,
    Verify,
}

match cmd {
    Command::Add { description, category } => {
        // description and category are extracted here, ready to use
        println!("Adding {} in {}", description, category);
    }
    Command::List => {
        println!("Listing all entries");
    }
    Command::Verify => {
        println!("Verifying chain");
    }
}

Matching on Option<T>

// entries.last() returns Option<&LogEntry>
match entries.last() {
    Some(entry) => {
        println!("Last entry: {}", entry.description);
    }
    None => {
        println!("No entries yet");
    }
}

Matching on Result<T, E>

match fs::read_to_string(path) {
    Ok(content) => {
        println!("Read {} bytes", content.len());
    }
    Err(e) => {
        println!("Error: {}", e);
    }
}

if let — single-case match

When you only care about one pattern, if let is cleaner:

// Full match — verbose
match entries.last() {
    Some(e) => println!("{}", e.hash),
    None => {}  // do nothing
}

// if let — cleaner
if let Some(e) = entries.last() {
    println!("{}", e.hash);
}

while let — looping until None

// Reading from an iterator while results are Some
let mut reader = BufReader::new(file);
let mut line = String::new();

while let Ok(bytes) = reader.read_line(&mut line) {
    if bytes == 0 break;  // EOF
    println!("Line: {}", line.trim());
    line.clear();
}

Guard clauses — matching with conditions

match entry.timestamp {
    // Only handle entries from the last 7 days
    t if (Utc::now() - t).num_days() < 7 => {
        println!("Recent entry");
    }
    // Handle older entries differently
    t if (Utc::now() - t).num_days() >= 7 => {
        println!("Archived entry");
    }
    _ => {}
}
🎯 Exhaustiveness checking

Rust's compiler checks that your match covers all possible cases. If you add a new enum variant, the compiler tells you everywhere you need to update your match statements. This prevents silent bugs where you forget to handle a new case.

PHASE 1 · Functional Programming

Closures & Iterators

Rust's closures and iterators are nearly identical to JavaScript's — with one key difference: Rust iterators are lazy and need .collect() to run.

Closures — like arrow functions

JavaScript
const double = x => x * 2;
const result = [1,2,3]
  .map(x => x * 2)
  .filter(x => x > 3);
Rust
let double = |x| x * 2;
let result: Vec<_> = vec![1,2,3]
  .iter()
  .map(|x| x * 2)
  .filter(|x| x > &3)
  .collect();

Iterator adapters are lazy

In JavaScript, .map() runs immediately. In Rust, it builds a pipeline that doesn't execute until you call .collect():

// This does NOTHING — iterator is lazy
let iter = vec![1, 2, 3]
    .iter()
    .map(|x| {
        println!("Processing {}", x);  // never prints!
        x * 2
    })
    .filter(|x| x > &3);

// This executes the pipeline
let result: Vec<_> = iter.collect();
// Now "Processing" lines print, once per element

Common iterator methods

MethodWhat it doesExample
.map(f)Transform each element|x| x * 2
.filter(f)Keep matching elements|x| x > 0
.filter_map(f)Filter + map in one|x| x.parse::<i32>().ok()
.for_each(f)Side effects, no return|x| println!("{}", x)
.find(f)Get first match|x| x == 5 → Option
.any(f)Does any match?|x| x > 10 → bool
.all(f)Do all match?|x| x > 0 → bool
.fold(init, f)Reduce/accumulate0, |acc, x| acc + x
.collect()Materialize into Vec/HashMapVec<i32>

filter_map — the power pattern

In vaultlog's storage layer, you'll read entries as JSON lines and skip any that fail to parse:

let entries: Vec<LogEntry> = BufReader::new(file)
    .lines()
    .filter_map(|line| {
        // Skip lines that can't be read
        let line = line.ok()?;
        // Skip lines that aren't valid JSON
        serde_json::from_str(&line).ok()
    })
    .collect();

filter_map takes a closure that returns Option — Some(value) is kept, None is discarded. Perfect for parsing and filtering in one step.

Closure borrowing — move vs borrow

let multiplier = 10;

// By default, closure borrows multiplier
let times_ten = |x| x * multiplier;
println!("{}", multiplier);  // ✓ still valid

// Use 'move' to take ownership
let times_ten = move |x| x * multiplier;
// multiplier is now owned by the closure, can't use it here
💡 Remember .collect()

Iterators in Rust are lazy — they're a sequence of transformations that hasn't run yet. Forgetting .collect() is one of the most common iterator mistakes. The compiler will tell you if a variable is unused, which helps catch this.

use sha2::{Digest, Sha256};
use crate::models::LogEntry;

pub fn compute_hash(entry: &LogEntry) -> String {
    let mut entry_copy = entry.clone();
    entry_copy.hash = "".to_string();
    
    let content = serde_json::to_string(&entry_copy)
        .expect("Failed to serialize");
    
    let mut hasher = Sha256::new();
    hasher.update(content.as_bytes());
    format!("{:x}", hasher.finalize())
}

pub fn verify_chain(entries: &[LogEntry]) -> bool {
    for i in 1..entries.len() {
        if entries[i].previous_hash != entries[i - 1].hash {
            return false;
        }
    }
    true
}

const LEDGER_FILE: &str = "ledger.ndjson";

pub fn append_entry(entry: &LogEntry) 
  -> Result<(), Box<dyn std::error::Error>>
{
    let mut file = OpenOptions::new()
        .create(true)
        .append(true)
        .open(LEDGER_FILE)?;

    let line = serde_json::to_string(entry)?;
    writeln!(file, "{}", line)?;
    Ok(())
}

pub fn read_all_entries() 
  -> Result<Vec<LogEntry>, Box<dyn std::error::Error>>
{
    let file = File::open(LEDGER_FILE)
        .unwrap_or_else(|_| {
            // If file doesn't exist, return empty vec
            panic!("ledger.ndjson not found")
        });

    let entries = BufReader::new(file)
        .lines()
        .filter_map(|line| line.ok())
        .filter_map(|line| serde_json::from_str(&line).ok())
        .collect();

    Ok(entries)
}

src/main.rs — complete CLI with Clap

mod ledger;
mod models;
mod storage;

use chrono::Utc;
use clap::{Parser, Subcommand};
use models::{EntryCategory, LogEntry};
use std::error::Error;
use uuid::Uuid;

#[derive(Parser)]
#[command(name = "vaultlog")]
#[command(about = "Immutable, hash-chained asset ledger", long_about = None)]
struct Cli {
    #[command(subcommand)]
    command: Commands,
}

#[derive(Subcommand)]
enum Commands {
    /// Add a new entry to the ledger
    Add {
        #[arg(short, long)]
        description: String,
        #[arg(short, long)]
        category: String,
    },
    /// List all entries
    List,
    /// Verify chain integrity
    Verify,
}

fn main() -> Result<(), Box<dyn Error>> {
    let cli = Cli::parse();

    match cli.command {
        Commands::Add { description, category } => {
            handle_add(description, &category)?;
        }
        Commands::List => {
            handle_list()?;
        }
        Commands::Verify => {
            handle_verify()?;
        }
    }

    Ok(())
}

fn handle_add(description: String, category_str: &str) 
  -> Result<(), Box<dyn Error>>
{
    let entries = storage::read_all_entries()
        .unwrap_or_default();

    let previous_hash = entries.last()
        .map(|e| e.hash.clone())
        .unwrap_or_else(|| "0000000000000000".to_string());

    let category = parse_category(category_str);

    let mut entry = LogEntry {
        id: Uuid::new_v4().to_string(),
        timestamp: Utc::now(),
        category,
        description,
        metadata: serde_json::json!({}),
        previous_hash,
        hash: "".to_string(),
    };

    entry.hash = ledger::compute_hash(&entry);
    storage::append_entry(&entry)?;

    println!("✓ Entry added: {}", entry.id);
    Ok(())
}

fn handle_list() -> Result<(), Box<dyn Error>> {
    let entries = storage::read_all_entries()?;

    if entries.is_empty() {
        println!("No entries yet.");
        return Ok(());
    }

    println!("\n{} entries:\n", entries.len());
    for (i, entry) in entries.iter().enumerate() {
        println!("[{}] {} | {}
          Category: {:?}",
            i + 1,
            entry.timestamp.format("%Y-%m-%d %H:%M"),
            entry.description,
            entry.category
        );
    }
    println!();

    Ok(())
}

fn handle_verify() -> Result<(), Box<dyn Error>> {
    let entries = storage::read_all_entries()?;

    if entries.is_empty() {
        println!("No entries to verify.");
        return Ok(());
    }

    if ledger::verify_chain(&entries) {
        println!("✓ Chain integrity verified — {} entries, all hashes valid", entries.len());
    } else {
        println!("✗ Chain broken! Data has been tampered with.");
    }

    Ok(())
}

fn parse_category(s: &str) -> EntryCategory {
    match s.to_lowercase().as_str() {
        "asset" => EntryCategory::Asset,
        "event" => EntryCategory::Event,
        "transfer" => EntryCategory::Transfer,
        "inspection" => EntryCategory::Inspection,
        _ => EntryCategory::Note,
    }
}

PHASE 2 · Async Foundations

async/await in Rust

Rust's async/await looks similar to JavaScript's, but it's built on a different foundation. Understanding this difference is crucial for web development.

JS async is built on promises

// JavaScript: automatic executor
async function fetchUser(id) {
    const res = await fetch(`/api/users/${id}`);
    return res.json();
}
// The browser/Node runtime automatically schedules this

Rust async needs a runtime

Rust's async/await is lazy — it doesn't run until a runtime (like Tokio) executes it. This gives you fine-grained control at the cost of explicitness:

// Cargo.toml
[dependencies]
tokio = { version = "1", features = ["full"] }
axum = "0.7"

// src/main.rs
#[tokio::main]  // macro that sets up Tokio runtime
async fn main() {
    println!("Async main running");
}

async fn returns a Future

In Rust, async fn doesn't execute immediately — it returns a Future, which is a lazy value that does something when awaited:

// This function is lazy — returns a Future
async fn fetch_entry(id: &str) -> String {
    sleep(Duration::from_secs(1)).await;
    format!("Entry {}", id)
}

// Calling the function doesn't execute it — just creates a Future
let future = fetch_entry("123");

// Await makes the runtime execute it
let result = future.await;  // Now it actually runs

Tokio tasks — lightweight concurrency

In Tokio, you spawn tasks instead of threads. Tasks are much cheaper:

#[tokio::main]
async fn main() {
    // Spawn 100 lightweight tasks
    let mut handles = vec![];

    for i in 0..100 {
        let handle = tokio::spawn(async move {
            println!("Task {}", i);
        });
        handles.push(handle);
    }

    // Wait for all to complete
    for handle in handles {
        handle.await.unwrap();
    }
}
🎯 Key difference from JS

JS async: automatically runs with the event loop. Rust async: lazy, needs a runtime (Tokio) to execute. You have explicit control — tasks don't start until spawned or awaited.

PHASE 2 · Web Framework

Axum Basics

Axum is a web framework built on Tokio. Handlers are async functions that return JSON, HTML, or custom responses. Routing is composed from handlers.

A minimal Axum server

use axum::{
    routing::get,
    Router, Json,
};
use serde_json::json;
use std::net::SocketAddr;
use tokio::net::TcpListener;

#[tokio::main]
async fn main() {
    // Define routes
    let app = Router::new()
        .route("/", get(root))
        .route("/health", get(health));

    // Bind and run
    let listener = TcpListener::bind("127.0.0.1:3000")
        .await
        .unwrap();

    println!("Listening on {}", listener.local_addr().unwrap());

    axum::serve(listener, app)
        .await
        .unwrap();
}

// Handler — just an async function
async fn root() -> Json<serde_json::Value> {
    json!({
        "message": "Welcome to vaultlog API"
    })
}

async fn health() -> Json<serde_json::Value> {
    json!({
        "status": "ok"
    })
}

Handlers are just async functions

A handler takes extractors as parameters and returns a response. Axum handles serialization/deserialization automatically:

// Handler with no parameters
async fn list_entries() -> Json<Vec<LogEntry>> {
    let entries = get_entries_from_storage();
    Json(entries)
}

// Handler with path parameter
async fn get_entry(Path(id): Path<String>) -> Json<LogEntry> {
    find_entry(&id)
}

// Handler with request body
async fn add_entry(Json(payload): Json<CreateEntryRequest>) -> Json<LogEntry> {
    let entry = create_new_entry(&payload.description, &payload.category);
    Json(entry)
}

Routing — connecting handlers to paths

let app = Router::new()
    // GET /entries → list_entries handler
    .route("/entries", get(list_entries))
    // POST /entries → add_entry handler
    .route("/entries", post(add_entry))
    // GET /entries/:id → get_entry handler
    .route("/entries/:id", get(get_entry))
    // GET /verify → verify_chain handler
    .route("/verify", get(verify_chain));
🎯 Handler anatomy

A handler is:

1. An async fn

2. Takes zero or more extractors as parameters

3. Returns a response type (Json, Html, String, custom Response)

Axum automatically deserializes request data into extractors and serializes return types into HTTP responses.

PHASE 2 · Concurrency & State

Shared State

In Axum, handlers run concurrently. To safely share data between them, you need to understand Arc (Atomic Reference Counting) and Mutex (Mutual Exclusion).

The problem: multiple handlers, one database

// Imagine we want all handlers to access the same database
// But handlers run concurrently — how do we safely share it?

let app = Router::new()
    .route("/entries", post(add_entry))
    .route("/entries", get(list_entries))
    .route("/verify", get(verify));

Arc<T> — shared ownership

Arc stands for "Atomic Reference Counting". It lets multiple owners share the same value:

// Without Arc — can't share
let db = Database::new();
// db is owned here. We can't pass it to multiple handlers.

// With Arc — shared ownership
let db = Arc::new(Database::new());
// Now db can be cloned and shared to many handlers
let db1 = db.clone();  // cheap clone, same data
let db2 = db.clone();  // another reference, same data

Mutex<T> — safe concurrent access

Mutex ensures only one task can access the data at a time:

// Without Mutex — NOT thread-safe
let count = 0;
// If two handlers try to read/write count simultaneously, undefined behavior

// With Mutex — thread-safe
let count = Mutex::new(0);

// To access the data, you must lock it
let mut guard = count.lock().await;  // waits if another task holds the lock
*guard += 1;  // now you can safely modify
drop(guard);  // lock is released here (or when guard goes out of scope)

Arc<Mutex<T>> — shared mutable state

Combine Arc and Mutex to share mutable data across async tasks:

#[tokio::main]
async fn main() {
    // Create an in-memory ledger shared across handlers
    let ledger = Arc::new(Mutex::new(Vec::<LogEntry>::new()));

    let app = Router::new()
        .route("/entries", post(add_entry))
        .route("/entries", get(list_entries))
        .with_state(ledger);

    // ... serve app
}

async fn add_entry(
    State(ledger): State<Arc<Mutex<Vec<LogEntry>>>>,
    Json(payload): Json<CreateEntryRequest>,
) -> Json<LogEntry> {
    // Lock the ledger to add an entry
    let mut ledger = ledger.lock().await;
    let entry = create_entry(&payload.description, &payload.category);
    ledger.push(entry.clone());
    Json(entry)
}
⚠ Holding locks across .await

Never hold a Mutex lock while awaiting. The lock should be acquired, modified, and released within a single synchronous section. Holding a lock across async boundaries can cause deadlocks.

mod models;
mod ledger;

use models::{EntryCategory, LogEntry};

#[derive(Serialize, Deserialize)]
struct CreateEntryRequest {
    description: String,
    category: String,
}

#[derive(Serialize, Deserialize)]
struct ListQuery {
    category: Option<String>,
    limit: Option<usize>,
}

#[derive(Serialize)]
struct VerifyResponse {
    valid: bool,
    message: String,
}

type SharedLedger = Arc<Mutex<Vec<LogEntry>>>;

#[tokio::main]
async fn main() {
    // Shared ledger state
    let ledger: SharedLedger = 
        Arc::new(Mutex::new(Vec::new()));

    // Define routes
    let app = Router::new()
        .route("/entries", post(add_entry))
        .route("/entries", get(list_entries))
        .route("/entries/:id", get(get_entry))
        .route("/verify", get(verify_chain))
        .with_state(ledger);

    // Run server
    let listener = TcpListener::bind("127.0.0.1:3000")
        .await
        .unwrap();

    println!("vaultlog API listening on {}", listener.local_addr().unwrap());

    axum::serve(listener, app)
        .await
        .unwrap();
}

// POST /entries — add a new entry
async fn add_entry(
    State(ledger): State<SharedLedger>,
    Json(payload): Json<CreateEntryRequest>,
) -> (StatusCode, Json<LogEntry>) {
    let mut entries = ledger.lock().await;

    let previous_hash = entries.last()
        .map(|e| e.hash.clone())
        .unwrap_or_else(|| "0000000000000000".to_string());

    let mut entry = LogEntry {
        id: Uuid::new_v4().to_string(),
        timestamp: Utc::now(),
        category: parse_category(&payload.category),
        description: payload.description,
        metadata: serde_json::json!({}),
        previous_hash,
        hash: "".to_string(),
    };

    entry.hash = ledger::compute_hash(&entry);
    entries.push(entry.clone());

    (StatusCode::CREATED, Json(entry))
}

// GET /entries — list all entries (with optional category filter)
async fn list_entries(
    State(ledger): State<SharedLedger>,
    Query(params): Query<ListQuery>,
) -> Json<Vec<LogEntry>> {
    let entries = ledger.lock().await;

    let filtered = entries
        .iter()
        .filter(|e| {
            if let Some(cat) = &params.category {
                format!("{:?}", e.category).to_lowercase() == cat.to_lowercase()
            } else {
                true
            }
        })
        .take(params.limit.unwrap_or(100))
        .cloned()
        .collect();

    Json(filtered)
}

// GET /entries/:id — get a single entry
async fn get_entry(
    State(ledger): State<SharedLedger>,
    Path(id): Path<String>,
) -> Result<Json<LogEntry>, StatusCode> {
    let entries = ledger.lock().await;
    
    entries
        .iter()
        .find(|e| e.id == id)
        .map(|e| Json(e.clone()))
        .Ok()
        .or(Err(StatusCode::NOT_FOUND))
}

// GET /verify — verify chain integrity
async fn verify_chain(
    State(ledger): State<SharedLedger>,
) -> Json<VerifyResponse> {
    let entries = ledger.lock().await;

    let valid = ledger::verify_chain(&entries);

    Json(VerifyResponse {
        valid,
        message: if valid {
            format!("Chain verified — {} entries", entries.len())
        } else {
            "Chain broken!".to_string()
        },
    })
}

fn parse_category(s: &str) -> EntryCategory {
    match s.to_lowercase().as_str() {
        "asset" => EntryCategory::Asset,
        "event" => EntryCategory::Event,
        "transfer" => EntryCategory::Transfer,
        "inspection" => EntryCategory::Inspection,
        _ => EntryCategory::Note,
    }
}

sqlx Basics

sqlx is an async SQL library with compile-time query checking. You write SQL, and sqlx verifies it against your database schema at compile time.

Setup with Supabase

Add to Cargo.toml:

sqlx = { version = "0.7", features = ["runtime-tokio-native-tls", "postgres"] }
dotenv = "0.15"
tokio = { version = "1", features = ["full"] }

Create .env:

DATABASE_URL=postgresql://user:password@host:5432/database

In Supabase, get your connection string from the "Connect" button in your project dashboard.

Connecting to the database

Running a simple query

use sqlx::Row;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let pool = create_pool().await?;

    // Execute a raw query
    let row = sqlx::query("SELECT NOW()")
        .fetch_one(&pool)
        .await?;

    let timestamp: chrono::DateTime<chrono::Utc> = row.get(0);

}

Querying into structs

#[derive(sqlx::FromRow)]
struct LogEntry {
    id: String,
    description: String,
    category: String,
    hash: String,
    previous_hash: String,
}

async fn fetch_all_entries(
    pool: &sqlx::postgres::PgPool
) -> Result<Vec<LogEntry>, sqlx::Error> {
    // sqlx::query_as automatically maps columns to struct fields
    sqlx::query_as::<_, LogEntry>(
        "SELECT id, description, category, hash, previous_hash
         FROM log_entries ORDER BY created_at DESC"
    )
    .fetch_all(pool)
    .await
}

Parameterized queries (preventing SQL injection)

// NEVER do this:
// let query = format!("SELECT * FROM entries WHERE id = '{}'", user_input);

// DO THIS instead (safe from SQL injection):
async fn find_entry_by_id(
    id: &str,
    pool: &sqlx::postgres::PgPool
) -> Result<LogEntry, sqlx::Error> {
    sqlx::query_as::<_, LogEntry>("SELECT * FROM log_entries WHERE id = $1")
        .bind(id)
        .fetch_one(pool)
        .await
}

The $1, $2, etc. are placeholders. .bind() safely parameterizes the query.

🎯 Compile-time checking

If you run cargo build with DATABASE_URL set and the database online, sqlx will verify your queries at compile time. Typos in column names and type mismatches become compile errors, not runtime panics.

PHASE 3 · Query Patterns

Query Building

Common patterns for inserting, updating, and fetching data. Every query is parameterized to prevent SQL injection.

Insert — adding an entry

#[derive(Serialize, Deserialize)]
struct NewLogEntry {
    description: String,
    category: String,
}

async fn insert_entry(
    entry: &NewLogEntry,
    hash: &str,
    previous_hash: &str,
    pool: &sqlx::postgres::PgPool,
) -> Result<String, sqlx::Error> {
    let id = uuid::Uuid::new_v4().to_string();

    sqlx::query(
        "INSERT INTO log_entries (id, description, category, hash, previous_hash, created_at)
         VALUES ($1, $2, $3, $4, $5, NOW())"
    )
    .bind(&id)
    .bind(&entry.description)
    .bind(&entry.category)
    .bind(hash)
    .bind(previous_hash)
    .execute(pool)
    .await?;

    Ok(id)
}

Select with filtering

async fn list_entries(
    category: Option<&str>,
    limit: i32,
    pool: &sqlx::postgres::PgPool,
) -> Result<Vec<LogEntry>, sqlx::Error> {
    let mut query = sqlx::query_as::<_, LogEntry>(
        "SELECT * FROM log_entries WHERE 1=1"
    );

    // Conditionally add WHERE clause
    if let Some(cat) = category {
        query = sqlx::query_as::<_, LogEntry>(
            "SELECT * FROM log_entries WHERE category = $1 LIMIT $2"
        )
        .bind(cat)
        .bind(limit);
    } else {
        query = sqlx::query_as::<_, LogEntry>(
            "SELECT * FROM log_entries LIMIT $1"
        )
        .bind(limit);
    }

    query.fetch_all(pool).await
}

Update

async fn update_entry(
    id: &str,
    new_description: &str,
    pool: &sqlx::postgres::PgPool,
) -> Result<u64, sqlx::Error> {
    let result = sqlx::query(
        "UPDATE log_entries SET description = $1 WHERE id = $2"
    )
    .bind(new_description)
    .bind(id)
    .execute(pool)
    .await?;

    Ok(result.rows_affected())
}

Transaction — multiple queries atomically

async fn add_entry_with_audit(
    entry: &NewLogEntry,
    pool: &sqlx::postgres::PgPool,
) -> Result<String, sqlx::Error> {
    // Start a transaction
    let mut tx = pool.begin().await?;

    // Insert entry
    let id = uuid::Uuid::new_v4().to_string();
    sqlx::query("INSERT INTO log_entries (id, description, category) VALUES ($1, $2, $3)")
        .bind(&id)
        .bind(&entry.description)
        .bind(&entry.category)
        .execute(&mut *tx)
        .await?;

    // Insert audit log
    sqlx::query("INSERT INTO audit_log (entry_id, action) VALUES ($1, 'created')")
        .bind(&id)
        .execute(&mut *tx)
        .await?;

    // Commit both at once, or both rollback on error
    tx.commit().await?;

    Ok(id)
}
💡 Always use parameters ($1, $2, etc.)

Never string-interpolate user input into SQL. Always use .bind() to pass parameters safely.

PHASE 3 · Schema Management

Migrations

Migrations are versioned SQL files that define your schema. sqlx runs them in order, ensuring your database stays in sync with your code.

Setting up migrations

// Install sqlx-cli (one-time)
cargo install sqlx-cli --no-default-features --features postgres

// Create initial migration
sqlx migrate add -r create_log_entries

This creates migrations/ directory with files like:

migrations/
├── 20240101120000_create_log_entries.up.sql
└── 20240101120000_create_log_entries.down.sql

Migration files for vaultlog

migrations/20240101120000_create_log_entries.up.sql:

-- Create the main log entries table
CREATE TABLE log_entries (
    id UUID PRIMARY KEY,
    description TEXT NOT NULL,
    category TEXT NOT NULL,
    hash VARCHAR(64) NOT NULL UNIQUE,
    previous_hash VARCHAR(64) NOT NULL,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Create indexes for fast querying
CREATE INDEX idx_category ON log_entries(category);
CREATE INDEX idx_created_at ON log_entries(created_at DESC);
CREATE INDEX idx_hash ON log_entries(hash);

migrations/20240101120000_create_log_entries.down.sql (rollback):

DROP TABLE IF EXISTS log_entries CASCADE;

Running migrations

// Run all pending migrations
sqlx migrate run --database-url $DATABASE_URL

// In Rust code, run migrations on startup:
#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    dotenv::dotenv().ok();

    let database_url = std::env::var("DATABASE_URL")?;

        .connect(&database_url)
        .await?;

    // Run migrations automatically
    sqlx::migrate!("./migrations")
        .run(&pool)
        .await?;

    println!("✓ Migrations complete");

    Ok(())
}

Commit migrations to git. Each migration is versioned. New developers or CI/CD can run sqlx migrate run to get the exact schema your code expects. No more "script to create the database" chaos.

src/handlers.rs — Axum handlers using database

use axum::{
    extract::{Path, State},
    http::StatusCode,
    Json,
};
use sqlx::postgres::PgPool;

use crate::{
    db, ledger, models::{CreateEntryRequest, LogEntry},
};

pub async fn add_entry(
    State(pool): State<PgPool>,
    Json(req): Json<CreateEntryRequest>,
) -> Result<(StatusCode, Json<LogEntry>), StatusCode> {
    let entries = db::get_all_entries(&pool)
        .await
        .map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)?;

    let previous_hash = entries.first()
        .map(|e| e.hash.clone())
        .unwrap_or_else(|| "0000000000000000".to_string());

    // Create temporary entry for hashing
    let mut temp_entry = LogEntry {
        id: uuid::Uuid::new_v4().to_string(),
        description: req.description.clone(),
        category: req.category.clone(),
        hash: "".to_string(),
        previous_hash: previous_hash.clone(),
        metadata: serde_json::json!({}),
        created_at: chrono::Utc::now(),
    };

    temp_entry.hash = ledger::compute_hash(&temp_entry);

    // Insert into database
    let entry = db::create_entry(&pool, &req, &temp_entry.hash, &previous_hash)
        .await
        .map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)?;

    Ok((StatusCode::CREATED, Json(entry)))
}

pub async fn list_entries(
    State(pool): State<PgPool>,
) -> Result<Json<Vec<LogEntry>>, StatusCode> {
    db::get_all_entries(&pool)
        .await
        .map(Json)
        .map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)
}

pub async fn get_entry(
    State(pool): State<PgPool>,
    Path(id): Path<String>,
) -> Result<Json<LogEntry>, StatusCode> {
    db::get_entry_by_id(&pool, &id)
        .await
        .map(Json)
        .map_err(|_| StatusCode::NOT_FOUND)
}

pub async fn verify_chain(
    State(pool): State<PgPool>,
) -> Result<Json<serde_json::Value>>, StatusCode> {
    let entries = db::get_all_entries(&pool)
        .await
        .map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)?;

    let valid = ledger::verify_chain(&entries);

    Ok(Json(serde_json::json!({
        "valid": valid,
        "count": entries.len(),
    })))
}
