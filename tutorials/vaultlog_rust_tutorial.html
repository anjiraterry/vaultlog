Orientation

The JS → Rust mental shift

You already think like a programmer. This guide rewires that thinking for Rust — using JavaScript and TypeScript as the translation layer throughout.

Why Rust feels strange at first

Every unusual piece of Rust syntax is solving a problem JavaScript handles invisibly. Once you see what it's solving, the syntax makes sense on its own.

JS/TS conceptRust equivalentKey difference
package.jsonCargo.tomlNear-identical concept
npmcargoAlso builds, runs, tests — one tool
TS interfacestructGets methods via impl
TS union typesenumEach variant can hold data
try/catchResult<T,E>Errors are values, not thrown
null/undefinedOption<T>Explicit "maybe nothing"
x => ...|x| ...Different syntax, same idea
.map().filter().map().filter().collect()Needs .collect() to run

Four things that make Rust different

1. Compiled, not interpreted. Your code becomes native machine code before it ever runs. Many JS runtime errors simply cannot exist in compiled Rust.

2. Static types — no surprises. Every value has a type the compiler knows at compile time. No "5" + 5 = "55" surprises.

3. No garbage collector. JavaScript's GC cleans up memory in the background. Rust uses ownership rules to free memory exactly when needed — zero runtime overhead.

4. Immutable by default. In JS, let is mutable. In Rust, let is immutable. You opt into mutation with let mut.

Key mindset

When Rust rejects your code, it's not being difficult — it's catching a bug you'd spend hours debugging at runtime. The compiler is your first line of defence. Its error messages often suggest exactly what to fix.

Day 1 · src/models.rs

Variables & types

Rust's type system is the foundation. Stricter than TypeScript, more explicit — but that strictness is what makes Rust safe and fast.

let is immutable by default

JavaScript
// let is MUTABLE in JS
let count = 0;
count = 1;  // ✓ fine

const name = "vault";
name = "x"; // ✗ runtime
Rust
// let is IMMUTABLE in Rust!
let count = 0;
count = 1;  // ✗ compile error!

// mut opts into mutability
let mut count = 0;
count = 1;  // ✓ fine
JS translation

Think of Rust's let as JS's const. Think of Rust's let mut as JS's let.

Numeric types

JS has one number type. Rust has many — it cares exactly how much memory a value takes and whether it can be negative:

Rust typeWhat it is
i32Signed 32-bit integer (default for integers)
u32Unsigned 32-bit (positive only, 0–4 billion)
i64 / u6464-bit variants, larger range
f6464-bit float (default for decimals)
usizePointer-sized unsigned — used for indices
booltrue or false

The two string types — this really matters

TypeWhat it isWhen to use
&strBorrowed reference to a string. Fixed-size. Usually string literals.Function params, strings you only read
StringOwned, heap-allocated string. Can grow, can be modified.Struct fields, strings built at runtime
JavaScript — one string type
const a = "vault";
function greet(name) {
  return `Hi ${name}`;
}
Rust — two string types
let a: &str = "vault";
let b: String = "vault".to_string();
fn greet(name: &str) -> String {
    format!("Hi {}", name)
}
Rule of thumb

Use String for struct fields. Use &str for function parameters where you only need to read. Convert: "hello".to_string() gives a String; &my_string gives a &str.

Structs & enums

These are how you model your data. Structs are TypeScript interfaces with methods. Rust enums are far more powerful than anything TypeScript has.

Structs — typed data containers

TypeScript interface
interface LogEntry {
  id: string;
  description: string;
  category: EntryCategory;
  previousHash: string;
  hash: string;
}
Rust struct
pub struct LogEntry {
    pub id: String,
    pub description: String,
    pub category: EntryCategory,
    pub previous_hash: String,
    pub hash: String,
}

pub is like export in JS — everything in Rust is private by default.

The #[derive] attribute — auto-generated code

#[derive(Serialize, Deserialize, Debug, Clone)]
pub struct LogEntry { /* ... */ }
Derive traitWhat it gives you
Serializestruct → JSON automatically
DeserializeJSON → struct automatically
Debugprintln!("{:?}", entry) works
Clone.clone() deep-copies the struct
Traits

A trait is Rust's version of an interface — it defines behaviour a type can have. When you #[derive(...)] one, Rust writes the implementation for you. When you need custom behaviour, you write it in an impl block.

Enums — sum types with real power

TypeScript enum (just constants)
enum Category {
  Asset = "ASSET",
  Event = "EVENT",
}
// Every value is just a string
Rust enum (variants)
pub enum EntryCategory {
    Asset,
    Event,
    Transfer,
    Inspection,
    Note,
}
// Used as: EntryCategory::Asset

Enums with data — where it gets interesting

Each variant can hold completely different data. This is how the CLI Commands enum works:

enum Commands {
    Add {
        description: String,  // holds two named fields
        category: String,
    },
    List,    // holds nothing
    Verify,  // holds nothing
}

Option<T> — Rust's null replacement

Rust has no null. Instead, a built-in enum:

match entries.last() {
    Some(entry) => println!("{}", entry.description),
    None        => println!("No entries yet"),
}
// Compiler forces you to handle BOTH cases — no null surprises
Day 2 · src/ledger.rs

Functions & methods

How to write functions, attach behaviour to structs with impl blocks, and understand Rust's implicit-return rule.

Defining functions

TypeScript
function greet(
  name: string
): string {
  return `Hello, ${name}!`;
}
Rust
fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
    // No semicolon = implicit return
}
Semicolons are semantic

In Rust, a semicolon turns an expression (which has a value) into a statement (which doesn't). The last line with no semicolon is the implicit return. Add a semicolon and you'll get: expected String, found (). The fix: remove the semicolon.

impl blocks — attaching methods to structs

In JS, methods live inside classes. In Rust, data (struct) and behaviour (impl) are deliberately separate:

JavaScript class
class LogEntry {
  constructor(desc) {
    this.description = desc;
  }
  getDesc() {
    return this.description;
  }
}
Rust struct + impl
struct LogEntry {
    description: String
}
impl LogEntry {
    pub fn get_desc(&self) -> &str {
        &self.description
    }
}

Three flavors of self

SyntaxMeaningJS equivalent
&selfBorrow read-only (most common)this — only reading fields
&mut selfBorrow mutably — can modify fieldsthis — writing fields
selfTake ownership (struct is consumed)No direct equivalent
no selfAssociated function (like static)static class method

compute_hash — the real vaultlog function

pub fn compute_hash(entry: &LogEntry) -> String {
    let content = serde_json::to_string(entry)
        .expect("Failed to serialize entry");

    let mut hasher = Sha256::new();
    hasher.update(content.as_bytes());
    format!("{:x}", hasher.finalize())
}

.expect("message") is the quick way to crash with a message if something fails. In production you'd use ? instead (chapter 5).

Day 2–3 · everywhere

Error handling

One of the biggest mental shifts from JS. Rust has no exceptions — errors are values your functions return explicitly.

Result<T, E> — Rust's try/catch replacement

Result is a built-in enum with two variants:

enum Result<T, E> {
    Ok(T),   // success — holds the value
    Err(E),  // failure — holds the error
}
JavaScript try/catch
async function readFile(path) {
  try {
    const data = fs.readFileSync(path);
    return data;
  } catch(err) {
    console.error(err);
  }
}
Rust Result
fn read_file(path: &str)
  -> Result<String, io::Error>
{
    let content = fs::read_to_string(path)?;
    Ok(content)
}

The ? operator — error propagation made easy

The ? at the end of a line is the most important operator for beginners to learn. It means: "if this returns Err, immediately return that error from my function. If it's Ok, unwrap and continue."

// Without ? — verbose
let file = OpenOptions::new().open(path);
let file = match file {
    Ok(f) => f,
    Err(e) => return Err(e),
};

// With ? — clean
let file = OpenOptions::new().open(path)?;

Box<dyn Error> — accepting any error type

In vaultlog's storage functions you'll see this return type:

pub fn append_entry(entry: &LogEntry)
    -> Result<(), Box<dyn Error>>
{
    let mut file = OpenOptions::new()
        .create(true)
        .append(true)
        .open("ledger.ndjson")?;

    let line = serde_json::to_string(entry)?;
    writeln!(file, "{}", line)?;
    Ok(())   // explicit "success, no value"
}
PatternWhen to use
?Propagate errors up — use this in almost every function
.expect("msg")Panic with message — fine in prototypes, avoid in libraries
.unwrap()Panic silently — only when you're certain it's Ok
match on ResultWhen you need to handle Ok and Err differently
The () unit type

() pronounced "unit" is Rust's "no meaningful value" type. Result<(), E> means "succeeds with nothing, or fails with an error" — like a void function that can fail.

Day 3 · the hardest part

Ownership & borrowing

The concept unique to Rust. There is no garbage collector — ownership rules determine exactly when memory is freed. This is what the borrow checker enforces.

Three ownership rules

  1. Every value in Rust has exactly one owner.
  2. There can only ever be one owner at a time.
  3. When the owner goes out of scope, the value is dropped (freed).
let s1 = String::from("hello");
let s2 = s1;  // s1 is MOVED into s2
println!("{}", s1);  // ✗ compile error! s1 no longer owns this
Why JS never does this

In JS, objects are reference-counted by the GC — multiple variables can "point" to the same data. Rust forbids this to eliminate entire classes of bugs (data races, use-after-free). The trade-off: you think about ownership explicitly.

Borrowing with & — read without owning

Instead of moving, you can lend a value temporarily:

fn compute_hash(entry: &LogEntry) -> String {
    // &LogEntry = "I borrow this, I don't own it"
    // The caller still owns `entry` after this call
    serde_json::to_string(entry).unwrap()
}

let entry = LogEntry { /* ... */ };
let hash = compute_hash(&entry);  // lend entry
println!("{:?}", entry);  // ✓ entry still valid — we only lent it

Borrowing rules

RuleWhat it means
Any number of &T (shared borrows)Many readers are fine simultaneously
Exactly one &mut T at a timeOne writer — no readers while writing
Cannot mix bothNo reading while something is being mutated

Clone — explicit deep copy

let s1 = String::from("hello");
let s2 = s1.clone();  // explicit copy — both are valid
println!("{} {}", s1, s2);  // ✓ both work

In vaultlog, #[derive(Clone)] on LogEntry lets you call .clone() on any entry whenever you need a second independent copy.

When you hit borrow checker errors

Read the error message carefully — Rust's compiler tells you exactly what the problem is and often suggests the fix. The most common fix is either: pass a reference (&entry instead of entry), or clone when you genuinely need two owners (entry.clone()).

Day 3 · src/storage.rs

Iterators & closures

Rust's iterator methods look a lot like JavaScript's array methods — .map(), .filter(), .for_each(). The main difference: they're lazy and need .collect().

Closures — like arrow functions

JavaScript
// Arrow function
const double = x => x * 2;
[1,2,3].map(x => x * 2);
Rust
// Closure — pipes instead of arrows
let double = |x| x * 2;
vec![1,2,3].iter().map(|x| x * 2);

Iterators are lazy — collect() runs them

In JS, .map() runs immediately. In Rust, iterator chains are lazy — they build up a pipeline and do nothing until you call .collect():

// JS — runs immediately
arr.filter(x => x > 0).map(x => x * 2);

// Rust — builds pipeline, runs only at .collect()
let result: Vec<i32> = arr
    .iter()
    .filter(|x| **x > 0)
    .map(|x| x * 2)
    .collect();  // ← this is where work happens

storage.rs — reading entries with iterators

pub fn read_all_entries()
    -> Result<Vec<LogEntry>, Box<dyn Error>>
{
    let file = File::open("ledger.ndjson")?;
    let reader = BufReader::new(file);

    let entries = reader
        .lines()
        .filter_map(|line| line.ok())       // skip error lines
        .filter_map(|line|                    // skip bad JSON
            serde_json::from_str(&line).ok())
        .collect();                           // → Vec<LogEntry>

    Ok(entries)
}

filter_map is like .filter().map() combined — it keeps only elements where the closure returns Some(value) and discards None. It's your go-to for fallible transformations.

Common iterator methods

Day 4 · src/main.rs

Pattern matching

match is Rust's supercharged switch statement. It's exhaustive — the compiler forces you to handle every possible case. You'll use it constantly.

Basic match

JavaScript switch
switch(category) {
  case "asset":
    doAsset(); break;
  case "event":
    doEvent(); break;
  default:
    doOther();
}
Rust match
match category {
    EntryCategory::Asset => {
        do_asset();
    }
    EntryCategory::Event => {
        do_event();
    }
    _ => do_other(),  // catch-all
}

Matching enums with data — the CLI dispatcher

This is the heart of vaultlog's main.rs. Each Commands variant holds different data, and match extracts it:

match cli.command {
    Commands::Add { description, category } => {
        // description and category are extracted here
        let entry = create_entry(description, category)?;
        storage::append_entry(&entry)?;
        println!("✓ Entry added");
    }
    Commands::List => {
        let entries = storage::read_all_entries()?;
        for entry in &entries {
            println!("{} | {}", entry.timestamp, entry.description);
        }
    }
    Commands::Verify => {
        let entries = storage::read_all_entries()?;
        if ledger::verify_chain(&entries) {
            println!("✓ Chain integrity verified");
        } else {
            println!("✗ Chain is broken!");
        }
    }
}

if let — matching a single variant

When you only care about one case, if let is cleaner than a full match:

// Full match — verbose when you only care about Some
match entries.last() {
    Some(e) => println!("{}", e.hash),
    None => {}  // do nothing
}

// if let — cleaner
if let Some(e) = entries.last() {
    println!("{}", e.hash);
}

The Clap CLI derive pattern

#[derive(Parser)]
#[command(name = "vaultlog")]
#[command(about = "Immutable hash-chained ledger")]
struct Cli {
    #[command(subcommand)]
    command: Commands,
}

#[derive(Subcommand)]
enum Commands {
    /// Add a new entry to the ledger
    Add {
        #[arg(short, long)]
        description: String,
        #[arg(short, long)]
        category: String,
    },
    /// List all entries
    List,
    /// Verify chain integrity
    Verify,
}

Clap reads those #[derive] and #[arg] annotations and generates a full CLI parser. The triple-slash comments (///) become help text.

Day 4–5 · complete data flow

Putting it all together

Here's the full vaultlog data flow from CLI input to verified ledger — every concept from every chapter in one picture.

Complete main.rs

mod models;
mod ledger;
mod storage;

use clap::Parser;
use chrono::Utc;
use models::{LogEntry, EntryCategory};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let cli = Cli::parse();

    match cli.command {
        Commands::Add { description, category } => {
            let entries = storage::read_all_entries()
                .unwrap_or_default();

            let previous_hash = entries.last()
                .map(|e| e.hash.clone())
                .unwrap_or_else(|| "0000000000000000".to_string());

            let mut entry = LogEntry {
                id: uuid::Uuid::new_v4().to_string(),
                timestamp: Utc::now(),
                category: parse_category(&category),
                description,
                metadata: serde_json::json!({}),
                previous_hash,
                hash: String::new(),
            };

            entry.hash = ledger::compute_hash(&entry);
            storage::append_entry(&entry)?;
            println!("✓ Added: {}", entry.id);
        }
        Commands::List => {
            let entries = storage::read_all_entries()?;
            for entry in &entries {
                println!("[{}] {} — {}",
                    entry.timestamp.format("%Y-%m-%d"),
                    entry.description,
                    entry.id);
            }
        }
        Commands::Verify => {
            let entries = storage::read_all_entries()?;
            if ledger::verify_chain(&entries) {
                println!("✓ Chain verified — {} entries", entries.len());
            } else {
                println!("✗ Chain broken — data was tampered with!");
            }
        }
    }
    Ok(())
}
